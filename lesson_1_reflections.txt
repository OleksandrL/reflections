How did viewing a diff between two versions of a file help you see the bug that
was introduced?

    1. Оскільки ми змінюємо робочий файл, ймовірно, що введений баг буде розміщуватися у
	зміненій частині файлу. Тому, порівнюючи стару (робочу) версію з новою (неробочою)
	можна виявити баг.

How could having easy access to the entire history of a file make you a more
efficient programmer in the long term?

    1. Якщо файл змінюєте тільки Ви, то історія може свідчити про розвиток Вашого уявлення
	відносно функцій, які забезпечує файл. Тому можна використати в майбутньому, щоб
	не повторювати помилок, а обрати одразу найбільш вигідний варіант. Якщо файл може
	змінити ще хтось, то історія дозволяє побачити, що, де і коли було змінено, і,
	якщо з'явився баг, то більш точно знати його розташування. Також можна робити
	відкати до більш ранніх робочих версій. Історія дозволяє швидкий перегляд змін
	файлу, що корисно при пошуці рішень подібних задач.
	
	2. Можна повернутися до більш ранніх версій, які задовольняли користувача. Можна пере-
	вірити, хто і які зміни зробив, порівнявши дві версії файла. Можна переглянути різні
	варіанти розв'язання задач. Проте тут робота пов'язана з одним файлом, що унеможлив-
	лює до певної міри взаємодію різних файлів і об'єднання їх в одну систему. Можна ви-
	користовувати будь-яку програму для зміни коду та підсвітку синтаксису.

What do you think are the pros and cons of manually choosing when to create a
commit, like you do in Git, vs having versions automatically saved, like Google
docs does?

    1. Переваги: дозволяє повертатися назад, якщо зміни не збережені; з файлами можна
	працювати локально.
	Недоліки: оновлення інформації на сервері потребує додаткових дій і часу, що
	викликає потребу у комунікації з колегами, які можуть працювати з цим же файлом,
	при збої втрачається прогрес.
	
	2. Переваги: затвердження змін повністю залежить від програміста, який на свій розсуд
	визначає степень зрілості файлу та зроблених змін для створення версії; дозволяє
	краще орієнтуватися в історії змін, оскільки містить інформацію про зміну, дату, 
	автора;
	Недоліки: при збої системи буде втрачений прогрес, якщо зміни не затверджені.

Why do you think some version control systems, like Git, allow saving multiple
files in one commit, while others, like Google Docs, treat each file separately?

    1. Тому що файли можуть взаємодіяти між собою: зміни в одному файлі можуть викли-
	кати зміни у іншому. Тому, змінивши і налаштувавши всі файли, можна їх затверджу-
	вати. У Доксах робота ведеться онлайн, тому необхідне постійне оновлення.
	
	2. Git створений для керування файлами з кодом, які пов'язані між собою.
	Оскільки код розбивається на декілька файлів - зміна одного з них може призвести до змін в іншому, тому
	необхідні зв'язок та синхронізоване оновлення вмісту файлів. Це дозволяє мати 
	більш ранню версію усього проекту, а не окремого файлу.

How can you use the commands git log and git diff to view the history of files?

    1. git log дозволяє побачити описану еволюцію файлу: що, де і коли було додано, чи
	виправлено тоді як git diff дозволяє з'ясувати як саме був змінений файл.
	
	2. git log --stat дозволяє побачити список версій проекту, який може містити декілька
	файлів. Мітка додає інформацію щодо файлів, у яких були проведені зміни, кількісні дані
	щодо змін. Можна отримати ID, щоб використати у команді git diff для порівняння змін у
	двох версіях проекту по файлах (виділяються рядки, які видалені, і які додані).

How might using version control make you more confident to make changes that
could break something?

    1. Система дозволяє завжди повернутися до старішої версії файлу, без помилок. Це і 
	забезпечує упевненість у роботі, адже зміни, які викликають помилки, можна відмінити.
	
	2. Впевненість у роботі пов'язана з можливістю відновлення робочої версії проекту.
	Вона дає змогу виявити першу несправну версію і подивитися, що саме було змінено.
	Це в свою чергу дозволяє виправити помилку і продовжити роботу над проектом. Система 
	дозволяє вводити значні зміни у проект і експериментувати з ним, оскільки завжди можна
	повернутися до робочої версії.
